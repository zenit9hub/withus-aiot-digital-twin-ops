# 에이전트설계

## 1. 문서 목적
본 문서는 핵심본체 샘플 프로젝트에서 사용하는 멀티에이전트 AI(Artificial Intelligence) 운영지원 구조의 `상위 개념`을 정의한다.  
이 문서는 역할 분리 원칙을 설명하는 공통 기준 문서이며, 실제 호출 구조와 실행 설계는 `엣지제어운영서버/AI연계.md`에서 다룬다.

## 2. 왜 공통설계에 두는가
- 멀티에이전트의 `역할 분리 원칙`은 시스템 전체가 공유하는 기준이다.
- 이 문서는 특정 도구나 호출 방식보다 `어떤 판단을 어떤 역할로 나눌 것인가`를 설명한다.
- 실제 호출 시점, 입력 생성, fallback 정책은 엣지제어운영서버 구현에 속하므로 하위 문서로 분리한다.

## 3. 공통 설계 원칙
- 멀티에이전트는 `제어`가 아니라 `운영지원`을 담당한다.
- 셧다운과 같은 안전 제어는 규칙 기반 자동화 로직이 담당한다.
- 에이전트는 원시 센서 전체가 아니라 요약된 상태와 이벤트를 입력으로 받는다.
- 각 에이전트의 책임은 겹치지 않게 유지한다.
- 결과는 최종 권고만이 아니라 판단 출처까지 함께 보여주어야 한다.

## 4. 역할 구조
핵심본체는 아래 3개 역할을 기본 구조로 사용한다.

설계 의도:
- `원인 추정`, `운영 가이드`, `최종 권고`를 분리해 설명 가능성과 재사용성을 높인다.

### 4.1 이상진단 에이전트
역할:
- 현재 이상 상태의 원인 후보를 추정한다.

하지 않는 일:
- 셧다운 여부 결정
- 최종 운영 명령 생성

### 4.2 운영지원 에이전트
역할:
- 운영자가 바로 실행할 수 있는 점검 순서와 대응 가이드를 제시한다.

하지 않는 일:
- 원인의 기술적 최종 판정
- 셧다운 명령 실행

### 4.3 종합판단 에이전트
역할:
- 이상진단 결과와 운영지원 결과를 종합해 최종 운영 권고를 생성한다.

하지 않는 일:
- 개별 센서 데이터 상세 분석
- 규칙 기반 제어 대체

## 5. 공통 입력과 출력 원칙
### 입력 원칙
- 센서 원시 데이터 전체를 직접 넣지 않는다.
- 상태 요약, 주요 이벤트, 설비 상태, 제어 이력 중심으로 입력한다.

### 출력 원칙
- 원인 후보, 점검 순서, 최종 권고처럼 사람이 바로 이해할 수 있는 형태로 정리한다.
- 결과는 `RULE`, `LOCAL_AI`, `AGENT_API` 같은 판단 출처와 함께 표시한다.

## 6. 로컬 AI와 외부 멀티에이전트의 관계
- 로컬 AI는 외부 연동이 어려운 상황에서 최소 운영지원 기능을 담당한다.
- 외부 멀티에이전트는 역할 분리 기반의 정교한 판단을 담당한다.
- 두 결과는 함께 보여줄 수 있지만, 반드시 구분하여 표시한다.
- 외부 API(Application Programming Interface) 실패 시에는 로컬 AI만으로 운영지원 기능을 유지한다.

## 7. 문서 경계
이 문서가 다루는 범위:
- 에이전트 역할 구조
- 책임 분리 원칙
- 공통 입력 및 출력 원칙

이 문서가 다루지 않는 범위:
- 호출 시점
- 입력 요약 생성 로직
- 외부 API 연결 방식
- fallback 실행 흐름
- 대시보드 반영 구현

위 항목은 `엣지제어운영서버/AI연계.md`에서 다룬다.
