# WBS(Work Breakdown Structure)

## 1. 문서 목적
본 문서는 핵심본체 샘플 프로젝트를 어떤 순서로 구체화하고 구현할지 정리한 실행 기준 문서다.  
목적은 `무엇을 만들 것인가`보다 `무엇을 먼저 확정할 것인가`를 분명하게 보여주는 데 있다.

## 2. 전체 진행 단계
1. 공통설계 확정
2. 설비환경시뮬레이터 상세설계
3. 엣지제어운영서버 상세설계
4. AI(Artificial Intelligence) 연계 설계
5. 구현 및 통합 검증

## 3. 단계별 작업
### 3.1 공통설계 확정
목적:
- 프로젝트 범위, 시나리오, 메시지 구조, 에이전트 역할 원칙을 공통 기준으로 정리한다.

주요 산출물:
- `02_WBS.md`
- `01_PRD.md`
- `03_데모시나리오.md`
- `04_메시지스키마.md`
- `05_에이전트설계.md`

### 3.2 설비환경시뮬레이터 상세설계
목적:
- 시뮬레이터의 상태 모델과 Tick 구조를 구현 가능한 수준으로 정리한다.

주요 산출물:
- `아키텍처_설비환경시뮬레이터.md`

### 3.3 엣지제어운영서버 상세설계
목적:
- 수집, 상태 판정, 제어, 운영 대시보드 반영 구조를 확정한다.

주요 산출물:
- `아키텍처_엣지제어운영서버.md`
- `Node-RED플로우.md`
- `제어규칙.md`
- `Dashboard.md`

### 3.4 AI 연계 설계
목적:
- 로컬 AI와 외부 멀티에이전트의 호출 시점, 입력 구조, 결과 반영 방식을 정리한다.

주요 산출물:
- `AI연계.md`

### 3.5 구현 및 통합 검증
목적:
- 시뮬레이터와 엣지제어운영서버를 실제로 연결하고 핵심 시나리오를 재현한다.

검증 대상:
- 정상 상태
- 냉각 제어 메시지 전달 시 회복 시나리오
- 냉각 제어 메시지 미전달 시 셧다운 시나리오
- 진동 경고 시나리오
- 진동 셧다운 시나리오

## 4. 단계별 핵심 완료 기준
- 1단계: 공통설계 문서 세트 확정
- 2단계: 시뮬레이터 상태 모델과 Tick 기준 확정
- 3단계: 엣지 수집, 제어, 운영 대시보드 구조 확정
- 4단계: 로컬 AI와 외부 멀티에이전트 연계 구조 확정
- 5단계: 핵심 데모 시나리오 재현과 통합 검증 완료

## 6. 문서화 원칙
- 상위 문서는 짧고 기준 중심으로 유지한다.
- 상세 문서는 컴포넌트별로 분리한다.
- 설계 문서는 바로 구현으로 내려갈 수 있는 수준까지 정리한다.
- 구현 문서는 이후 별도 저장소 또는 구현 문서로 분리 관리한다.
