# AI연계

## 1. 문서 목적
본 문서는 `엣지제어운영서버`가 로컬 AI와 외부 멀티에이전트를 어떻게 호출하고, 결과를 어떻게 반영하는지 정의한다.  
이 문서는 실행 설계 문서이며, 역할 분리의 상위 원칙은 `../00_공통설계/05_에이전트설계.md`를 따른다.

## 2. AI 연계 역할
엣지제어운영서버는 아래 역할을 수행한다.

- AI 호출 시점 결정
- AI 입력 요약 객체 생성
- 로컬 AI 호출
- 외부 멀티에이전트 호출
- 결과 병합
- 운영 대시보드 반영
- API(Application Programming Interface) 실패 시 fallback 수행

## 3. 호출 원칙
- AI는 `제어`가 아니라 `운영지원`을 담당한다.
- 셧다운 여부는 규칙 기반 제어 로직이 먼저 결정한다.
- AI 호출은 주기 호출보다 `상태 변화` 또는 `이벤트 발생` 기반 호출이 적절하다.
- 원시 센서값 전체가 아니라 요약된 상태를 전달한다.

## 4. 호출 트리거
권장 트리거는 아래와 같다.

- `WARNING` 이상 상태 진입
- `SHUTDOWN` 발생
- `VIBRATION_WARNING` 발생
- `VIBRATION_CRITICAL` 발생
- `COOLING_COMMAND_SENT` 또는 `COOLING_COMMAND_SKIPPED` 후 요약 보고가 필요한 경우
- `RECOVERY` 발생 후 요약 보고가 필요한 경우

권장 초기 호출 매트릭스:
- `WARNING`: 로컬 AI 우선 호출, 필요 시 외부 멀티에이전트 호출
- `VIBRATION_WARNING`: 로컬 AI 우선 호출
- `SHUTDOWN`: 로컬 AI와 외부 멀티에이전트 모두 호출
- `RECOVERY`: 요약 보고가 필요할 때만 호출

## 5. 입력 요약 객체 생성
AI 입력은 아래 정보를 포함하는 것이 적절하다.

- 현재 `runtimeState`
- 현재 `healthState`
- `shutdownFlag`
- 최근 온도 및 진동 요약
- 주요 이벤트 목록
- 최근 냉각 제어 이력
- 최근 추세 요약

입력 생성 원칙:
- 한 번의 호출에 필요한 최소 정보만 포함한다.
- 같은 시점 데이터는 `sampleId` 기준으로 조합한다.
- 최근 상태는 시간 구간 요약으로 전달한다.

## 6. 로컬 AI 연계
### 6.1 역할
- 외부 연동이 어려운 상황에서 최소 운영지원 기능 제공
- 상태 요약, 원인 후보, 점검 가이드 생성

### 6.2 호출 시점
- WARNING 이상 상태에서 우선 호출 가능
- 외부 API 실패 시 fallback으로 호출

### 6.3 출력 반영
- `summary`
- `possibleCause`
- `nextAction`

## 7. 외부 멀티에이전트 연계
### 7.1 역할 구조
- 이상진단
- 운영지원
- 종합판단

### 7.2 호출 방식
- 단일 호출 내 역할 분리형 구조를 사용할 수 있다.
- 또는 역할별 순차 호출 후 결과를 종합할 수 있다.

### 7.3 출력 반영
- 원인 후보
- 점검 우선순위
- 최종 운영 권고
- 판단 출처

## 8. 결과 병합 방식
엣지제어운영서버는 아래 순서로 결과를 정리하는 것이 적절하다.

1. 규칙 기반 제어 결과 확정
2. 로컬 AI 결과 수신
3. 외부 멀티에이전트 결과 수신
4. 운영 대시보드 통합 상태 객체에 반영

병합 원칙:
- 규칙 기반 제어 결과를 우선한다.
- AI 결과는 제어를 덮어쓰지 않는다.
- 로컬 AI와 외부 멀티에이전트 결과는 구분해서 표시한다.

## 9. fallback 정책
### 외부 API 실패
- 로컬 AI만으로 운영지원 기능 유지
- 규칙 기반 제어와 운영 대시보드는 정상 유지

### 로컬 AI 실패
- 외부 멀티에이전트 결과만 반영 가능
- 둘 다 실패해도 규칙 기반 제어는 유지

### 전체 AI 실패
- 규칙 기반 제어와 이벤트 기록만으로 운영
- 운영 대시보드에는 `AI unavailable` 상태를 표시할 수 있다.

## 10. 운영 대시보드 반영 기준
운영 대시보드에는 아래를 구분해 표시한다.

- 규칙 기반 상태 판정 결과
- 로컬 AI 결과
- 외부 멀티에이전트 결과
- 최종 운영 권고
- 판단 출처

상세 화면 구성 기준은 `Dashboard.md`에서 다룬다.

## 11. 고도화 항목
아래 항목은 초기 MVP 이후 확장 대상으로 본다.

- 에어컨 `status` 기반 응답 확인
- 응답 없음 또는 timeout 판단
- 냉각 시스템 가동 중 온도 비회복 경고
- 제어 실패 메시지 자동 생성

## 12. 구현 권장사항
- AI 입력 생성은 별도 `function` 노드 또는 서비스로 분리한다.
- 호출 결과는 원본과 요약 결과를 분리 저장하는 것이 좋다.
- API 타임아웃과 재시도 정책은 짧고 단순하게 시작한다.
- 중요 상태 변화가 없을 때는 불필요한 AI 호출을 줄인다.
