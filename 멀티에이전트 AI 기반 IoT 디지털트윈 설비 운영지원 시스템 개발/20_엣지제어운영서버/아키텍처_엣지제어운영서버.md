# 아키텍처_엣지제어운영서버

## 1. 문서 목적
본 문서는 핵심본체 샘플 프로젝트에서 사용하는 `엣지제어운영서버`의 구조와 역할을 정의한다.  
초기 구현은 Node-RED/FlowFuse 기반으로 하며, 센서 수집, 상태 판정, 제어, AI(Artificial Intelligence) 연계, 운영 대시보드 표시를 담당한다.

## 2. 엣지제어운영서버 역할
엣지제어운영서버는 아래 역할을 수행한다.

- MQTT(Message Queuing Telemetry Transport) 메시지 수집
- 같은 시점 데이터 조합
- 온도 및 진동 상태 판정
- 냉각 제어 및 셧다운 수행
- 이벤트 생성
- AI 입력 요약 객체 생성
- 로컬 AI 및 외부 멀티에이전트 API(Application Programming Interface) 호출
- 운영 대시보드 상태 객체 생성

핵심은 `수집`, `판정`, `제어`, `운영지원`을 한 곳에서 연결하되, `안전 제어`와 `AI 운영지원`의 책임을 분리하는 것이다.

## 3. 외부 인터페이스
### 3.1 구독 토픽
- `factory/room-01/conveyor-01/status`
- `factory/room-01/temp-01/sensor`
- `factory/room-01/vib-01/sensor`

### 3.2 발행 토픽
- `factory/room-01/conveyor-01/control`
- `factory/room-01/aircon-01/control`

### 3.3 내부 생성 객체
- 센서 스냅샷 객체
- 이벤트 객체
- 규칙 기반 제어 결과 객체
- AI 입력 요약 객체
- 운영 대시보드 통합 상태 객체

## 4. 데이터 처리 흐름
1. MQTT 메시지를 구독한다.
2. `sampleId`를 기준으로 온도, 진동, 설비 상태를 조합한다.
3. 규칙 기반 로직으로 현재 상태를 판정한다.
4. 시나리오에 따라 냉각 제어 메시지를 전달하거나 전달하지 않는다.
5. 필요 시 셧다운 제어를 발행한다.
6. 이벤트를 생성한다.
7. AI 입력 요약 객체를 만든다.
8. 로컬 AI와 외부 멀티에이전트 API를 호출한다.
9. 결과를 운영 대시보드에 반영한다.

## 5. 스냅샷 조합 원칙
- 온도, 진동, 설비 상태는 같은 `sampleId`로 묶는다.
- `sampleId`가 없는 예외 상황이 있으면 짧은 시간 윈도우 기준으로 보조 조합할 수 있다.
- 조합에 실패한 불완전 데이터는 제어 판단보다 표시용 상태로만 제한하는 것이 바람직하다.

권장 초기 정책:
- `sampleId` 조합 대기 시간은 `1 tick` 또는 `약 2초` 이내로 시작한다.
- 온도, 진동, 설비 상태 3개가 모두 모이면 제어 판단에 사용한다.
- 일부만 도착한 스냅샷은 운영 대시보드 표시에는 사용할 수 있지만, 제어 판단에는 사용하지 않는다.

## 6. 상태 판정과 제어 규칙
### 6.1 온도 규칙
- `45도 이상`: WARNING, 냉각 제어 시작 구간
- `50도 이상`: CRITICAL
- `50도 이상이 일정 시간 지속`: SHUTDOWN

### 6.2 진동 규칙
- `0.70 이상`: `VIBRATION_WARNING`
- `0.85 이상`: `VIBRATION_CRITICAL`, 즉시 SHUTDOWN

### 6.3 냉각 제어 규칙
- 온도 경고 구간 진입 시 냉각 제어를 수행한다.
- 정상 시나리오에서는 냉각 제어 메시지를 시뮬레이터로 전달한다.
- 냉각 실패 시나리오에서는 시뮬레이터로 제어 메시지를 전송하지 않으므로 냉각 효과가 반영되지 않는다.
- 초기 MVP에서는 에어컨 `status`를 읽어 실패를 판정하지 않는다.

구현 메모:
- 시나리오 분기 지점은 `aircon/control` 발행 직전 블록으로 두는 것이 단순하다.
- 즉, 상태 판정 이후 `COOLING_COMMAND_SENT`와 `COOLING_COMMAND_SKIPPED`를 나누고, 그 결과에 따라 실제 `mqtt out` 발행 여부를 결정한다.

## 7. 이벤트 생성 기준
권장 이벤트는 아래와 같다.

- `WARNING`
- `SHUTDOWN`
- `RECOVERY`
- `COOLING_COMMAND_SENT`
- `COOLING_COMMAND_SKIPPED`
- `VIBRATION_WARNING`
- `VIBRATION_CRITICAL`

이벤트는 센서 입력 그대로 사용하는 것이 아니라, 엣지제어운영서버가 상태 판정 결과를 바탕으로 생성한다.

## 8. AI 연계 구조
### 8.1 로컬 AI
- 최근 상태 요약을 입력으로 받는다.
- 원인 후보와 점검 가이드를 생성한다.
- 외부 API 실패 시 최소 운영지원 기능을 유지한다.

### 8.2 외부 멀티에이전트 API
- 이상진단, 운영지원, 종합판단 역할로 분리한다.
- 상태가 WARNING 이상이거나 주요 이벤트가 발생했을 때 호출하는 구조가 적절하다.

### 8.3 역할 분리 원칙
- AI는 셧다운 여부를 결정하지 않는다.
- 셧다운은 규칙 기반 제어 로직이 담당한다.
- AI는 원인 후보, 점검 순서, 운영 권고를 생성한다.

## 9. 운영 대시보드 구성 기준
운영 대시보드에는 아래 항목이 표시되어야 한다.

- 현재 온도
- 현재 진동
- 설비 상태
- 최근 냉각 제어 명령 상태
- 최근 이벤트 로그
- 규칙 기반 제어 결과
- 로컬 AI 결과
- 외부 멀티에이전트 결과
- 판단 출처

상세 화면 구성 기준은 `Dashboard.md`에서 다룬다.

## 10. 시뮬레이터와의 역할 분리
- 설비환경시뮬레이터: 상태 계산과 센서 발행 담당
- 엣지제어운영서버: 상태 판정, 이벤트 생성, 제어 명령 담당
- AI 계층: 운영지원 판단 담당

즉, 시뮬레이터는 `현재 상태를 만든다`, 엣지제어운영서버는 `현재 상태를 해석하고 제어한다`, AI는 `현재 상태에 대한 운영 권고를 만든다`.

## 11. 고도화 항목
아래 항목은 초기 MVP 이후 확장 대상으로 본다.

- 에어컨 `status` 기반 응답 확인
- 응답 timeout 기반 실패 판정
- 냉각 시스템 가동 중 온도 비회복 경고
- 냉각 시스템 이상 메시지 자동 생성

## 12. 구현 권장사항
- Node-RED 플로우는 `수집`, `조합`, `규칙`, `제어`, `AI`, `운영 대시보드` 단위로 구분한다.
- 경고 상태와 셧다운 상태를 같은 플로우에서 처리하더라도, 조건 블록은 분리하는 것이 좋다.
- AI 호출은 주기 호출보다 `상태 변화` 또는 `이벤트 발생` 기반 호출이 더 적절하다.
- 외부 API 실패 시에도 운영 대시보드와 규칙 기반 제어는 정상 동작해야 한다.
